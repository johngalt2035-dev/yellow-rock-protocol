# Contributing to Yellow Rock Protocol

Welcome! We appreciate your interest in improving the Yellow Rock Protocol. This project benefits from contributions across multiple disciplines -- communications research, behavioral psychology, legal practice, and software development.

## How to Contribute

### Issues

- **Bug reports**: Something in the protocol documentation is incorrect or unclear
- **Feature requests**: New scenarios, templates, or protocol extensions
- **Research suggestions**: New citations, evidence, or corrections to existing claims

### Pull Requests

1. Fork the repository
2. Create a feature branch (`git checkout -b feature/your-feature-name`)
3. Make your changes
4. Test your changes (ensure all markdown renders correctly, verify links)
5. Submit a PR with a clear description of what changed and why

### Discussion

Open an issue for discussion before making large changes. This helps avoid duplicate work and ensures alignment with the project's direction.

## Research Contributions

We welcome contributions from psychologists, communications researchers, legal practitioners, mediators, and related professionals. The protocol's credibility depends on accurate, well-sourced claims.

### Guidelines for Research Contributions

- All factual claims must be linked to primary sources with full citations
- Clearly distinguish between RCT-validated findings, clinical/experiential evidence, and anecdotal reports
- When citing research, note the study's scope and limitations
- Do not overstate the strength of evidence -- if a finding is preliminary or context-dependent, say so
- New citations should include scope notes (see `docs/RESEARCH.md` Citation Index for format)

## Truthfulness Standard

This project maintains a strict truthfulness standard:

- **No unsubstantiated claims**: Every factual assertion must cite a primary source
- **Honest limitations**: If evidence is anecdotal or lacks RCT validation, state that explicitly
- **No exaggeration**: Do not claim the protocol "guarantees" outcomes or is "proven" without qualification
- **Correct attribution**: Credit original authors and frameworks accurately
- **Context matters**: Note when research from one domain (e.g., personal relationships) is being applied to another (e.g., business communications)

## Code of Conduct

- **Professional**: Maintain a professional, respectful tone in all interactions
- **Respectful**: Treat all contributors with respect regardless of background or experience level
- **Evidence-based**: Ground discussions in evidence, not opinion or personal experience alone
- **Constructive**: Offer constructive feedback; explain why something should change, not just that it should

## Issue Templates

### Bug Report

```
**Description**: [What is incorrect or broken?]
**Location**: [File and section]
**Expected behavior**: [What should it say/do?]
**Actual behavior**: [What does it currently say/do?]
```

### Feature Request

```
**Description**: [What feature or addition are you proposing?]
**Use case**: [What problem does this solve?]
**Evidence basis**: [Any supporting research or citations?]
```

### Research Suggestion

```
**Citation**: [Full citation in APA format]
**Relevance**: [Which section(s) of the protocol does this inform?]
**Evidence strength**: [RCT / clinical / experiential / anecdotal]
**Scope notes**: [What does this source cover and what are its limitations?]
```

## Pull Request Process

1. **Fork** the repository to your account
2. **Branch** from `main` with a descriptive name (e.g., `research/add-gottman-2015`, `feature/workplace-scenarios`)
3. **Make changes** following the project's style and standards
4. **Verify** all markdown renders correctly and all links work
5. **Submit** a PR with:
   - Clear title describing the change
   - Description of what changed and why
   - Citations for any new factual claims
   - Note if this is a breaking change to existing protocol sections
6. **Respond** to review feedback constructively
